# 附件与 Docs 生成测试用例

## 1. 目的与范围

本文覆盖从附件输入到 Docs 生成、编辑、构建和预览的核心质量路径。

- 作品类型：`docs`
- 默认模板：`fumadocs-docs`
- 覆盖范围：附件校验、内容提取、Brief、Docs 初次生成、编辑重建、失败恢复与数据安全。
- 不覆盖范围：发布、灰度、回滚与跨项目权限管理。

## 2. 通用前置条件

1. 测试环境已部署 Runtime、对象存储和 `fumadocs-docs` 模板。
2. 测试账号具备创建 Docs 项目、上传附件、确认 Brief、发起生成和编辑的权限。
3. 模型服务、附件解析服务、构建服务和预览服务可用。
4. 测试执行者可查看项目详情、Run 活动/SSE、构建结果、预览页面和审计日志。

## 3. 测试数据

| 编号 | 文件 | 用途 |
| --- | --- | --- |
| F-01 | `product-guide.md`：包含产品简介、安装、快速开始、FAQ 四个章节 | 正常 Docs 生成与编辑 |
| F-02 | `oversized.pdf`：大于系统允许的最大上传大小 | 超限校验 |
| F-03 | `corrupted.pdf`：文件格式声明为 PDF，但内容损坏 | 解析失败恢复 |
| F-04 | `unsupported.exe`：不在允许格式列表内 | 格式校验 |
| F-05 | `internal-roadmap.md`：含内部项目代号及模拟敏感 token | 可追溯与日志脱敏 |

## 4. 测试用例

### TC-01：Markdown 附件生成 Docs

**优先级：P0**

**目标：** 验证真实 Markdown 附件可完成“上传 → Brief → Docs → 构建 → 预览”闭环。

**测试步骤：**

1. 创建一个项目，项目名称填写“产品使用指南”，类型选择 `Docs`。
2. 上传 `F-01 product-guide.md`，观察上传页的文件名、类型、大小和处理状态。
3. 提交内容源，等待系统完成附件解析与 Brief 生成。
4. 在 Brief 中核对目标用户、章节层级、内容假设和缺失信息；确认 Brief。
5. 发起生成，持续查看 Run 活动/SSE，直至生成和构建完成。
6. 打开 Draft Preview 与当前预览，检查首页、安装、快速开始、FAQ 是否都可访问。

**预期结果：**

- 附件被保存为不可变 `ContentSource`，项目详情可查看其文件名、类型和处理状态。
- Brief 明确包含 Docs 项目类型、目标受众和合理的内容层级。
- 生成的 Fumadocs Docs 具有可用导航、首页和对应正文页面。
- `project.build` 成功，生成已提升的预览版本；浏览器可正常访问当前预览。

### TC-02：不合法附件在生成前被拦截

**优先级：P0**

**目标：** 验证不支持、超限或损坏的文件不会进入生成链路。

**测试步骤：**

1. 进入创建 Docs 项目的内容上传页面，确认页面已展示支持格式和最大文件大小。
2. 上传 `F-04 unsupported.exe`。
3. 重新进入上传页面，上传 `F-02 oversized.pdf`。
4. 再上传 `F-03 corrupted.pdf`，等待解析结果。
5. 每次失败后检查项目内容源列表、Brief 列表、Run 列表和活动日志。

**预期结果：**

- 不支持格式和超限文件在客户端与服务端均被拒绝，并返回可理解的错误原因。
- 损坏文件在解析阶段返回明确错误，不被误判为可用内容。
- 三种失败均不会创建可执行的生成 Run，不会启动 Build，也不会产生可访问的预览。
- 已失败的上传不会污染后续正常附件的生成流程。

### TC-03：附件内容可追溯，日志不泄露敏感信息

**优先级：P1**

**目标：** 验证包含内部资料的附件在处理、生成和审计过程中的可追溯性与脱敏。

**测试步骤：**

1. 创建 Docs 项目，上传 `F-05 internal-roadmap.md`。
2. 完成 Brief 确认并生成 Docs 预览。
3. 在项目详情检查 ContentSource 元数据、来源、处理状态及关联 Brief/Run。
4. 查询 Run 活动/SSE、应用运行日志和附件访问审计记录。
5. 在预览中核对文档内容来自附件，但不额外暴露原始文件路径、对象存储地址或模拟 token。

**预期结果：**

- 项目内可将附件、Brief、Run、构建和预览版本串联追溯。
- 日志和 SSE 不输出附件全文、原始对象存储地址或模拟敏感 token。
- 附件、生成代码和预览均通过内部受控服务访问；没有外部公开地址或外发记录。
- 审计日志至少记录附件访问、生成发起和预览访问的主体、时间与结果。

### TC-04：Docs 编辑后重建并更新预览

**优先级：P0**

**目标：** 验证附件生成的 Docs 支持基于现有内容增量编辑、重建和预览更新。

**前置条件：** TC-01 已成功执行，并记录已提升版本 A 的版本标识与当前预览地址。

**测试步骤：**

1. 在项目对话中输入：“将安装章节改为快速开始，并新增 FAQ 页面；保留现有内容和导航风格。”
2. 提交编辑请求，查看 Run 活动，确认系统进入编辑与构建流程。
3. 检查变更后的 Docs 源文件：既有 MDX 内容应被修改，并新增 FAQ 内容/路由和导航项。
4. 等待 `project.build` 完成与新预览版本提升。
5. 刷新记录的当前预览地址，依次访问“快速开始”和“FAQ”。

**预期结果：**

- 编辑基于现有 Docs 源文件完成，不会回退为全新模板或丢失未修改章节。
- 生成的新版本 B 通过构建，且版本号与版本 A 不同。
- 当前预览 URL 保持不变，但页面内容、导航及 FAQ 已更新。
- Run 的事件顺序中，成功构建与 `preview.updated` 均出现在完成状态之前。

### TC-05：附件解析失败后可补充或替换并恢复生成

**优先级：P1**

**目标：** 验证系统能安全处理“可上传但无法形成有效 Docs 结构”的附件，并支持恢复。

**测试步骤：**

1. 创建 Docs 项目并上传 `F-03 corrupted.pdf`，提交生成 Brief。
2. 等待解析失败或 Brief 提示内容缺失，记录错误信息。
3. 验证原始附件仍可在项目中查看，且此时无法确认 Brief 或发起正式 Build。
4. 补充 Prompt：“请生成面向开发者的产品使用文档，至少包含安装、快速开始和常见问题。”并替换为 `F-01 product-guide.md`。
5. 重新生成并确认 Brief，发起 Docs 生成和构建。
6. 检查新 Run、生成结果和预览页面。

**预期结果：**

- 系统清楚说明解析失败或缺失内容的原因，并保留原始输入供排查。
- 未形成有效且已确认的 Brief 前，系统禁止正式 Build。
- 用户可替换附件或补充 Prompt 后重试；失败 Run 与新 Run 可区分且关联关系可追溯。
- 恢复后的生成结果与 TC-01 一致：Docs 可构建、可预览，且包含安装、快速开始和 FAQ 等约定内容。

## 5. 通过准则

1. TC-01、TC-02、TC-04 必须全部通过，方可认为附件驱动的 Docs 主链路达到 P0 可用。
2. TC-03、TC-05 必须通过，方可认为该链路满足基础的安全追溯与失败恢复要求。
3. 每个成功用例保留项目 ID、ContentSource ID、Brief ID、Run ID、版本 ID、SSE 记录和浏览器截图，作为验收证据。
