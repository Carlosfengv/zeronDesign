# 视觉优先的 AI React 网站与文档站生成平台 P0 User Stories

## 1. 文档目的

本文从用户视角定义视觉优先的 AI React 网站与文档站生成平台 P0 用户故事和验收标准。
P0 的产品目标接近 v0.dev 的 React 前端创作体验，不把 Lovable 式全栈应用能力作为当前
完成条件。

配套实施方案：`2026-07-19-visual-react-runtime-capability-implementation-plan.md`。

P0 需要完成以下核心闭环：

```text
创建项目
-> 输入需求 / 上传 Markdown / 添加视觉参考
-> AI 分析并提出必要问题
-> AI 推荐站点类型、Design Profile 和模型
-> AI 整理页面结构与内容
-> 用户确认内容方案
-> AI 生成网站
-> 用户预览并通过自然语言修改
-> 用户确认发布版本
-> 发布并获得固定 URL
-> 继续编辑草稿并重新发布
```

P0 的核心原则是：用户只需要描述目标，AI 负责推荐和整理；关键节点由用户确认；所有成功修改可恢复；发布过程不能破坏已有线上版本。

## 2. 产品目标

作为用户，我希望通过自然语言、Markdown 和视觉参考创建 React 网站或文档站，由 AI
完成需求理解、内容整理、视觉探索和前端生成；我可以在生成前确认内容与设计方向，在
生成后通过可视预览和自然语言持续修改，最终将确认版本发布到固定 URL。

## 3. P0 范围

P0 包含：

- 自然语言需求输入。
- Markdown 文件上传与解析。
- 图片或 Figma Frame 视觉参考输入。
- 项目自动保存与恢复。
- AI 澄清关键需求。
- 推荐站点类型、Design Profile 和生成模型。
- 内容整理、来源标记和用户确认。
- 通用 `next-app` React 前端模板；Fumadocs Docs 继续用于文档站。
- 真实截图进入模型视觉上下文，并支持参考图与候选画面的视觉比较。
- 受控组件注册表、图标和基础图片资产生成或导入。
- 常驻 Draft Preview Session 和 HMR 快速反馈。
- 响应式网站生成与交互预览。
- 有范围控制的自然语言修改。
- 基础版本记录、撤销和恢复。
- 草稿与线上版本隔离。
- 通过 Ingress 发布到固定 URL。
- 重新发布和发布失败保护。

P0 暂不包含自定义域名、私有站点、多人协作、代码导出、Git 同步、完整版本差异对比、
SEO 和访问统计。

Lovable 式全栈能力属于后续独立 Epic，不阻断本 P0，包括：Server Actions、用户数据库、
认证、后端 API、Secret/环境变量管理、第三方业务集成和有状态服务部署。P0 的 `next-app`
使用受控的前端与静态导出契约，不允许模型把上述全栈能力作为隐含依赖。

## 4. Epic A：创建项目与输入需求

### US-P0-01 创建生成项目

作为用户，我希望创建一个新项目并描述自己想要构建的网站，以便开始 AI 生成流程。

验收标准：

- 用户可以创建新项目。
- 系统为项目生成默认名称，用户可以修改。
- 用户可以不预先选择网站或文档站，直接输入需求。
- 项目创建后进入需求输入阶段。
- 项目拥有唯一标识。

### US-P0-02 输入自然语言需求

作为用户，我希望通过自然语言描述网站用途、目标用户和预期效果，以便 AI 理解我要构建什么。

验收标准：

- 支持多行纯文本输入。
- 用户可以描述网站目标、受众、页面、内容、功能和风格。
- 输入区域提供简单示例或引导问题。
- 用户可以在提交前编辑输入。
- 文本输入与当前项目关联并保存。

建议的引导问题：

- 你希望创建什么？
- 这个网站主要给谁使用？
- 你希望包含哪些页面？
- 有没有特别希望突出展示的内容？

### US-P0-03 上传 Markdown 内容

作为用户，我希望上传 Markdown 文件作为网站内容，以便 AI 在已有内容基础上生成网站或文档站。

验收标准：

- 支持上传一个或多个 `.md` 文件。
- 支持同时使用自然语言和 Markdown 文件。
- 系统能够解析标题、正文、列表、表格、代码块、链接和图片引用等常见 Markdown 内容。
- 上传后显示文件名称和解析状态。
- 用户可以删除或重新上传文件。
- 文件解析失败时说明原因。
- 单个文件解析失败不会影响其他已经成功上传的文件。
- 用户提交需求前，可以查看系统识别出的基本内容结构。

### US-P0-03A 添加视觉参考

作为用户，我希望上传截图、参考图片或选择 Figma Frame，以便 AI 不只根据文字猜测设计方向。

验收标准：

- 支持 PNG、JPEG、WebP 和已授权的 Figma Frame。
- 每个视觉参考保存为不可变 Design Source Artifact，并记录来源、尺寸、哈希和创建时间。
- 用户可以指定主参考，并说明希望继承的布局、色彩、组件或视觉氛围。
- 系统向模型提供真实像素内容，不得只提供文件名、URI、哈希或图片描述。
- 参考不可读取时显示明确错误，不静默降级为“已参考”。
- 用户可以删除或替换尚未冻结到 Run 的视觉参考。

### US-P0-04 自动保存和继续编辑

作为用户，我希望系统自动保存项目进度，以便离开后可以继续之前的工作。

验收标准：

- 自动保存用户输入、上传文件和当前配置。
- 保存状态对用户可见。
- 用户重新进入项目后可以继续之前的流程。
- 页面刷新或临时断开不会导致已保存内容丢失。
- 尚未生成的网站仍可以作为草稿继续编辑。

## 5. Epic B：AI 理解需求与推荐配置

### US-P0-05 AI 分析用户意图

作为用户，我希望 AI 综合分析文字需求和 Markdown 内容，以便形成对项目目标的准确理解。

AI 需要识别：

- 网站目标。
- 目标受众。
- 推荐站点类型。
- 核心内容。
- 可能需要的页面。
- 内容之间的关系。
- 设计倾向。
- 缺失、冲突或不明确的信息。

验收标准：

- AI 同时分析文字输入和附件内容。
- AI 不得无提示地忽略已上传内容。
- 不同输入之间存在冲突时，系统指出冲突。
- AI 输出简短的需求理解摘要。
- 用户可以纠正 AI 的理解。

### US-P0-06 AI 提出必要的澄清问题

作为用户，我希望 AI 在关键信息缺失或存在冲突时提出问题，以避免生成方向错误。

验收标准：

- AI 只询问会明显影响结果的关键问题。
- 避免一次提出大量问题。
- 对不影响主要结果的信息，AI 可以使用合理默认值。
- 用户可以回答问题，也可以选择使用 AI 推荐值。
- 用户回答后，AI 更新需求理解。
- 用户可以纠正之前的回答。

可能需要澄清的内容包括：

- 网站面向的主要受众。
- 普通网站还是文档站。
- 是否需要多个页面。
- 品牌或产品名称。
- Markdown 内容之间的冲突。
- 缺失的核心页面内容。

### US-P0-07 推荐站点类型与前端模板

作为用户，我希望 AI 根据需求推荐普通 React 网站或文档站及其模板，以便不理解框架差异也能正确开始。

验收标准：

- AI 推荐普通 React 网站或文档站中的一种。
- 普通网站默认使用 `next-app`，文档站默认使用 `fumadocs-docs`。
- `next-app` 是普通网站和通用 React 创作的默认模板。
- 系统简要解释推荐原因。
- 用户可以接受推荐。
- 用户可以手动切换站点类型。
- 切换后系统重新调整建议的页面结构。
- 用户确认前不会开始生成。

### US-P0-08 选择 Design Profile

作为用户，我希望查看并选择预期的设计风格，以便生成结果符合自己的视觉偏好。

验收标准：

- AI 根据需求推荐一个 Design Profile。
- 每个 Design Profile 至少包含名称、简短描述、视觉预览和适用场景。
- 用户可以接受推荐或选择其他 Design Profile。
- 当前选择状态清晰可见。
- Design Profile 会影响颜色、字体、排版、布局、组件和间距等视觉规则。
- 后续修改默认继续遵循当前 Design Profile。
- 用户可以在生成前更换 Design Profile。

### US-P0-09 选择生成模型

作为用户，我希望系统推荐一个适合的生成模型，同时允许我手动选择，以便在不理解模型差异时也能快速开始。

验收标准：

- 系统提供一个默认推荐模型。
- 用户不手动选择时使用推荐模型。
- 用户可以查看当前可用模型。
- 模型至少展示名称、生成质量倾向和速度倾向。
- 用户可以手动切换模型。
- 项目记录实际使用的模型。
- 开始生成前显示最终选定模型。

## 6. Epic C：内容整理与确认

### US-P0-10 AI 整理站点结构和内容

作为用户，我希望 AI 将原始输入整理成适合网站或文档站的内容结构，以便生成结果清晰、完整且易于浏览。

AI 整理结果包括：

- 网站名称和简介。
- 目标受众。
- 导航结构。
- 页面列表和页面层级。
- 每个页面的用途和摘要。
- 页面正文或内容大纲。
- 缺失内容。
- AI 建议补充的内容。

验收标准：

- 普通网站和文档站采用不同的信息架构。
- AI 保留用户提供的重要事实。
- AI 不得将推测内容描述为已确认事实。
- 内容整理结果可以被用户编辑。
- 修改页面结构后，关联内容能够同步调整。

### US-P0-11 标记内容来源和 AI 新增内容

作为用户，我希望知道内容来自原始输入、AI 改写还是 AI 新增，以便判断内容是否可信。

验收标准：

- 系统至少区分用户原始内容、AI 整理或改写内容、AI 新增或推测内容、待补充内容。
- AI 新增的重要事实性内容必须明确标记。
- 不确定或冲突的内容进入待确认区域。
- 用户可以接受、编辑或删除 AI 新增内容。
- 用户确认后，内容状态更新为已确认。
- 不得静默发布未经确认的关键推测内容。

### US-P0-12 确认内容方案

作为用户，我希望在生成网站前确认页面结构和主要内容，以避免直接生成错误结果。

确认内容分为三个层级：

1. 网站目标和受众。
2. 导航及页面结构。
3. 每个页面的摘要和待确认内容。

验收标准：

- 默认展示页面结构和页面摘要，不强制用户阅读全部正文。
- 用户可以展开查看和编辑完整内容。
- 用户可以直接编辑，也可以通过自然语言要求 AI 修改。
- 系统集中展示冲突内容、缺失内容和 AI 新增内容。
- 用户可以确认整个内容方案。
- 存在未解决的关键冲突时，系统提示用户处理。
- 只有用户明确确认后才能开始生成。
- 内容方案发生重大变化后，需要重新确认。

## 7. Epic D：网站生成

### US-P0-13 根据确认方案生成网站

作为用户，我希望 AI 根据已确认的内容、Design Profile 和模型生成可运行的网站。

验收标准：

- 生成必须基于最近一次确认的内容方案。
- 生成结果包含完整的页面和导航。
- 网站在桌面端和移动端可以正常显示。
- Markdown 内容中的常见元素能够正确呈现。
- 生成过程中显示当前状态。
- 生成完成后自动进入预览。
- 生成结果与当前项目关联并保存为一个版本。

### US-P0-13A 生成受控的 Next.js React 项目

作为用户，我希望普通网站生成在统一的 React 工程契约上，以便组件、交互和后续修改具有一致基础。

验收标准：

- Runtime 注册并版本化 `next-app` Template，框架为 Next.js App Router + React + TypeScript。
- Template 至少提供 `app/`、`components/ui/`、样式 Token、全局样式和受控静态导出配置。
- 生成页面、导航、表单状态、Tabs、菜单和主要 CTA 使用真实 React 组件与事件处理。
- Desktop 与 375px Mobile 均通过渲染、横向溢出、Console 和基础无障碍检查。
- Build、Preview、Source Snapshot、Artifact Manifest 和不可变 Version 与现有 Website/Docs 契约一致。
- P0 禁止 Server Actions、Route Handlers、用户数据库、认证和运行时 Secret；检测到时 Build 必须失败并给出明确指导。

### US-P0-13B 使用真实组件和视觉资产

作为用户，我希望 AI 使用一致的组件、图标和图片资产，而不是生成占位视觉，从而获得可继续完善的设计结果。

验收标准：

- AI 可以搜索和添加受控组件注册表中的组件，并记录组件来源与版本。
- 图标来自允许的图标库；缺失的图片资产可以通过受控生成或用户上传补齐。
- 不得用 Emoji、占位方块、ASCII、随意内联 SVG 或纯 CSS 图形冒充要求中的可见资产。
- 资产生成或导入时记录来源、尺寸、哈希、使用页面和可追溯信息。
- 组件和资产不能绕过 Template Mutation Policy、依赖策略和构建验证。

### US-P0-14 查看生成进度和处理失败

作为用户，我希望知道 AI 当前正在做什么，并在生成失败时可以安全重试。

验收标准：

- 生成过程至少展示准备内容、生成页面、构建网站和准备预览等状态。
- 生成失败时使用容易理解的语言说明原因。
- 用户可以重新尝试生成。
- 重试不会丢失需求、附件、内容方案和配置。
- 失败的生成不会覆盖上一个成功版本。
- 用户可以从最近一个安全步骤继续，不必重新创建项目。

## 8. Epic E：预览与自然语言修改

### US-P0-15 预览生成结果

作为用户，我希望在发布前浏览生成的网站，以便检查内容、页面和视觉效果。

验收标准：

- 提供可交互的网站预览。
- 用户可以访问站内所有已生成页面。
- 至少支持桌面端和移动端预览。
- 导航和核心链接可以正常使用。
- 显示当前预览对应的版本。
- 明确说明当前内容尚未发布或与线上版本不同。

### US-P0-15A 获得视觉审查和参考图比较

作为用户，我希望 AI 真正查看候选页面并与视觉参考比较，以便发现构图、层级和视觉一致性问题。

验收标准：

- Runtime 在 Desktop 与 Mobile 视口生成真实候选截图，并把图片内容提供给具备视觉能力的模型。
- 有主参考时，系统在相同视口和状态下比较参考图与候选图。
- 视觉审查至少覆盖布局、间距、字体、颜色、边框、圆角、图片裁切、内容密度和视觉层级。
- 审查结果绑定 `candidateVersionId + screenshotId + sourceRevision`，旧截图不能覆盖新候选。
- 视觉比较输出结构化 Finding 和证据，但不得只依赖一个不透明总分决定是否发布。
- 模型或视觉通道不可用时必须明确标记 `visual_review_unavailable`，不能伪装为通过。

### US-P0-15B 使用快速 Draft Preview Session

作为用户，我希望小范围修改快速出现在预览中，以便获得接近 v0.dev 的视觉迭代节奏。

验收标准：

- `next-app` Build/Edit Run 可以启动受控的常驻 Dev Session，并通过 Runtime Preview Proxy 访问。
- 文本和 CSS 修改从 source commit 到 iframe Ready 的 P50 不超过 1 秒。
- 组件或页面结构修改从 source commit 到 iframe Ready 的 P95 不超过 5 秒。
- 每次更新携带单调递增的 `sourceRevision`，过期结果不能覆盖新结果。
- 编译错误显示 Overlay 并保留上一成功画面；修复后复用同一 Session 恢复。
- HMR 超时后自动回退到完整 iframe reload，但 Draft 不自动成为 Promoted Version。

### US-P0-16 通过自然语言修改网站

作为用户，我希望通过自然语言修改网站的文字、样式、布局和页面结构，以便像 vibe coding 一样持续完善网站。

支持的修改范围包括：

- 修改文字。
- 修改颜色、字体和间距。
- 调整布局。
- 修改组件。
- 新增或删除页面。
- 修改导航结构。
- 调整移动端表现。
- 调整全站视觉风格。

验收标准：

- 用户可以连续提出多轮修改需求。
- AI 能结合当前项目和当前版本理解需求。
- AI 识别修改针对的页面或区域。
- 用户可以基于预览中选中的元素或区域发起修改，系统把稳定的页面、组件和 DOM 定位信息加入 Edit 上下文。
- 修改完成后更新预览。
- 未被要求修改的内容应尽量保持不变。
- 修改成功后形成新的可恢复版本。
- 修改失败不会破坏上一个成功版本。

### US-P0-17 控制修改范围

作为用户，我希望在 AI 修改之前知道将影响哪些内容，以避免一句话意外改变整个网站。

验收标准：

- AI 识别修改属于局部修改、页面级修改或全站修改。
- 对文案等低风险局部修改，可以直接执行。
- 删除页面、修改导航结构、更换全站 Design Profile、全站布局调整和大规模重写内容等高影响修改，在执行前需要用户确认。
- 确认信息中展示预计影响的页面和区域。
- Runtime 持久化结构化 Edit Impact Plan，至少包含 scope、目标文件/组件、目标页面和风险等级。
- 用户可以取消或调整修改指令。

### US-P0-18 撤销修改和恢复版本

作为用户，我希望能够撤销不满意的修改，以便安全尝试不同方案。

验收标准：

- 每次成功生成或修改后创建一个版本。
- 用户可以撤销最近一次修改。
- 用户可以查看基础版本记录，包括时间和修改摘要。
- 用户可以恢复到之前的成功版本。
- 恢复操作不会删除其他历史版本。
- 当前线上版本需要明确标记。
- 失败操作不创建可发布版本。

P0 暂不要求复杂的逐行差异对比，但必须支持基本版本恢复。

## 9. Epic F：确认与发布

### US-P0-19 确认待发布版本

作为用户，我希望在发布前确认具体版本和访问地址，以避免发布错误内容。

验收标准：

- 发布前展示待发布版本、网站名称、页面数量、URL、当前 Design Profile，以及它与当前线上版本的关系。
- 用户需要显式确认发布。
- 发布针对确定的版本，不受后续草稿修改影响。
- 如果确认后继续修改，需要重新确认新版本。
- 用户可以取消发布。

### US-P0-20 发布并获得固定 URL

作为用户，我希望将网站发布到固定 URL，以便持续分享和访问。

URL 可以采用：

```text
https://<project-slug>.<domain>
```

或：

```text
https://<domain>/<project-slug>
```

验收标准：

- 发布成功后返回固定访问 URL。
- URL 通过 Ingress 路由到对应项目。
- 用户可以复制访问链接。
- 用户可以在首次发布前确认或修改 URL slug。
- URL 冲突时，系统提示用户更换或生成可用替代值。
- 后续重新发布时固定 URL 保持不变。
- P0 发布的网站默认为公开访问，系统在发布前明确提示用户。
- 访问 URL 能够正常加载当前线上版本。

### US-P0-21 草稿与线上版本隔离

作为用户，我希望发布后继续修改网站，但不立即影响线上内容。

验收标准：

- 发布成功后，该版本被标记为线上版本。
- 用户后续修改产生新的草稿版本。
- 草稿修改不会改变固定 URL 当前展示的内容。
- 项目界面明确区分当前草稿版本和当前线上版本。
- 只有再次确认并发布后，线上版本才会更新。
- 用户离开后重新进入项目时，仍能看到草稿和线上版本状态。

### US-P0-22 安全地重新发布

作为用户，我希望将新版本发布到原有 URL，并在发布失败时继续保留旧网站。

验收标准：

- 重新发布继续使用原固定 URL。
- 发布过程中旧线上版本保持可访问。
- 新版本构建成功后才切换线上版本。
- 发布失败时不影响旧版本。
- 失败后显示原因并允许重试。
- 系统记录最后一个成功发布版本。
- 用户可以恢复并重新发布最后一个成功版本。

## 10. 项目状态

| 状态 | 用户看到的含义 |
|---|---|
| `Draft` | 正在输入需求或编辑项目 |
| `Analyzing` | AI 正在分析需求 |
| `Awaiting Answers` | 等待用户回答关键问题 |
| `Preparing Content` | AI 正在整理内容 |
| `Awaiting Content Approval` | 等待确认站点结构和内容 |
| `Generating` | 正在生成网站 |
| `Generation Failed` | 生成失败，可以重试 |
| `Draft Preview` | 常驻开发预览已就绪，可以快速修改 |
| `Visual Reviewing` | AI 正在查看候选截图或与参考图比较 |
| `Awaiting Edit Approval` | 高影响修改已规划，等待用户确认范围 |
| `Preview` | 可以预览和继续修改 |
| `Awaiting Publish Approval` | 当前版本等待发布确认 |
| `Publishing` | 正在发布，旧线上版本不受影响 |
| `Published` | 已发布并拥有固定 URL |
| `Publish Failed` | 发布失败，旧线上版本仍然可用 |

## 11. P0 关键产品规则

1. 用户不需要理解技术细节也能开始，站点类型、Design Profile 和模型均提供推荐默认值。
2. 未经用户确认的内容方案不能进入网站生成。
3. AI 新增或推测的重要内容必须明确标记。
4. 每次成功生成或修改必须产生可恢复版本。
5. 高影响修改必须在执行前展示影响范围。
6. 草稿修改不能直接影响线上网站。
7. 新版本发布成功前，旧线上版本始终保持可访问。
8. 项目固定 URL 在重新发布后保持不变。
9. P0 只支持公开发布，但必须在发布前明确告知用户。
10. 所有失败操作都必须保留最近一次成功结果和用户原始输入。
11. 普通网站默认使用 `next-app`，它承载 P0 通用 React 创作能力。
12. 视觉审查必须让模型获得真实图片内容，URI、哈希和 DOM 指标不能替代视觉输入。
13. P0 生成物必须遵守前端静态发布边界，不隐式引入全栈运行时能力。
14. Draft 快速反馈与 Promoted/Published 可信版本分离，HMR 成功不等于版本晋升。

## 12. P0 完成标准

当用户能够完整完成以下流程时，可以认为 P0 主线成立：

- 输入自然语言需求并上传 Markdown。
- 可以添加截图或 Figma Frame 作为视觉参考。
- AI 理解需求并提出必要问题。
- AI 推荐站点类型、Design Profile 和生成模型。
- AI 整理内容并标记内容来源。
- 用户确认页面结构和内容方案。
- AI 使用 `next-app` 或 `fumadocs-docs` 生成可交互预览的网站。
- AI 能查看 Desktop/Mobile 候选截图，并在存在参考图时完成结构化视觉比较。
- 文本、样式和常见结构修改通过 Draft Preview Session 获得快速反馈。
- 页面使用可追溯的真实组件、图标和视觉资产，不以占位视觉冒充完成。
- 用户通过自然语言完成多轮修改。
- 高影响修改在执行前展示结构化影响范围并获得一次性确认。
- 用户能够撤销或恢复修改。
- 用户确认并发布指定版本。
- 用户获得一个固定 URL。
- 用户继续修改草稿时不影响线上网站。
- 重新发布失败时，原有网站仍然正常访问。
